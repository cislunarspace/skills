# ADR 格式

ADR 存放在 `docs/adr/`，使用顺序编号：`0001-slug.md`、`0002-slug.md`，以此类推。

`docs/adr/` 目录按需创建，只在第一个 ADR 需要时才建。

## 模板

```md
# {决策的简短标题}

{1-3 句话：背景是什么、决定了什么、为什么。}
```

就这些。一个 ADR 可以只有一段话。价值在于记录**做了**某个决策以及**为什么**，不在于填模板。

## 可选段落

只在真正有价值时才加。多数 ADR 不需要。

- **Status** 前置信息（`proposed | accepted | deprecated | superseded by ADR-NNNN`）：决策会被重新审视时有用
- **考虑过的选项**：被否决的替代方案值得记住时
- **后果**：有不明显的下游影响需要指出时

## 编号

扫描 `docs/adr/` 找到最大编号，加一。

## 何时提议 ADR

以下三条必须同时满足：

1. **难以撤销**：事后改变主意的成本很大
2. **缺乏上下文会令人困惑**：未来的读者看到代码会想"他们到底为什么这么做？"
3. **源于真正的权衡**：存在真实的替代方案，你因特定理由选了其中一个

如果决策容易撤销，跳过，大不了改回来。如果不令人困惑，没人会问为什么。如果没有真正的替代方案，没什么可记的，"我们做了显而易见的事"而已。

### 什么样的决策值得记录

- **架构形态**。"我们用 monorepo。""写模型是事件溯源的，读模型投影到 Postgres。"
- **上下文之间的集成模式**。"Ordering 和 Billing 通过领域事件通信，不是同步 HTTP。"
- **带锁定效应的技术选型**。数据库、消息总线、认证提供商、部署目标。不是每个库，只是换掉需要一个季度的那些。
- **边界和范围决策**。"客户数据归 Customer 上下文所有，其他上下文只按 ID 引用。"显式的"不"和显式的"是"同样有价值。
- **对显而易见路径的刻意偏离**。"我们用手写 SQL 而非 ORM，因为 X。"任何合理读者会假设相反的情况。这些记录能阻止下一个工程师去"修正"刻意为之的设计。
- **代码中不可见的约束**。"因为合规要求，我们不能用 AWS。""因为合作方 API 契约，响应时间必须低于 200ms。"
- **非显而易见的否决**。如果你考虑过 GraphQL 最终因微妙的原因选了 REST，记下来，否则六个月后又会有人提议 GraphQL。
