# Domain Modeling

在设计过程中主动构建和打磨项目的领域模型。这是一项**主动**纪律：挑战术语、构造边界场景、在术语和决策成型的当下立刻写进术语表。（仅仅**读取** `CONTEXT.md` 获取词汇不是这个技能的职责，任何技能都能做。这个技能用于**改变**模型，而非消费模型。）

## 文件结构

多数仓库只有一个上下文：

```
/
├── CONTEXT.md
├── docs/
│   └── adr/
│       ├── 0001-event-sourced-orders.md
│       └── 0002-postgres-for-write-model.md
└── src/
```

如果根目录存在 `CONTEXT-MAP.md`，说明仓库有多个上下文。map 指向每个上下文的位置：

```
/
├── CONTEXT-MAP.md
├── docs/
│   └── adr/                          ← 系统级决策
├── src/
│   ├── ordering/
│   │   ├── CONTEXT.md
│   │   └── docs/adr/                 ← 上下文专属决策
│   └── billing/
│       ├── CONTEXT.md
│       └── docs/adr/
```

按需创建文件：有内容可写时才创建。没有 `CONTEXT.md`，就在第一个术语确定时创建。没有 `docs/adr/`，就在第一个 ADR 需要时创建。

## 会话过程中

### 对照术语表挑战

当用户使用的术语与 `CONTEXT.md` 中的已有定义冲突时，立刻指出。"你的术语表把'取消'定义为 X，但你似乎在说 Y，到底是哪个？"

### 锐化模糊语言

当用户使用模糊或歧义的词时，提出一个精确的规范术语。"你说'账户'，指的是 Customer 还是 User？这是两个不同的东西。"

### 讨论具体场景

当讨论领域关系时，用具体场景来压测。构造场景来探索边界情况，迫使用户精确界定概念之间的边界。

### 与代码交叉验证

当用户陈述某事的运作方式时，检查代码是否一致。发现矛盾就指出："你的代码取消的是整个 Order，但你刚说支持部分取消，哪个对？"

### 实时更新 CONTEXT.md

术语确定时，当场更新 `CONTEXT.md`，不要攒着。用 [CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md) 中的格式。

`CONTEXT.md` 中不应有任何实现细节。不要把它当作规格、草稿本或实现决策的仓库。它是术语表，仅此而已。

### 谨慎提议 ADR

只有以下三条同时满足时，才提议创建 ADR：

1. **难以撤销**：事后改变主意的成本很大
2. **缺乏上下文会令人困惑**：未来的读者会想"他们为什么这么做？"
3. **源于真正的权衡**：存在真实的替代方案，你因特定理由选了其中一个

三条缺一条，就跳过 ADR。用 [ADR-FORMAT.md](./ADR-FORMAT.md) 中的格式。
