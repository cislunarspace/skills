# Argument Construction

帮用户构建完整的论证：观点、理由、证据、承认与回应、论据。

## 步骤

### 1. 提炼观点

让用户用一句话说清观点，检查：
- 是概念性（让读者相信什么）还是实际性（让读者做什么）？
- 是否足够具体？（避免模糊词汇如"一些"、"很多"）
- 是否有重要意义？（读者接受后需要改变多少信念）

如果观点太模糊，让用户提供更多细节。

### 2. 组织理由

让用户提供2-3个支持观点的理由，检查：
- 每个理由是否能用"因为"连接到观点？
- 理由之间是否有逻辑关系？
- 理由是否勾勒出报告的轮廓？

用分镜表帮用户排列理由的顺序。

### 3. 寻找证据

对每个理由，问用户：
- 有什么证据支持这个理由？
- 证据来源是什么？是否可靠？
- 证据是否充分、有代表性？

如果证据不足，建议用户回到 `/source-collection`。

### 4. 预想反对意见

让用户扮演批评者，问自己：
- 读者会质疑什么？
- 有哪些替代方案我没考虑？
- 有哪些反例我需要解释？

选择最重要的2-3个反对意见，准备回应。

### 5. 补充论据

如果读者可能质疑理由和观点的关联：
- 用"如果X，则Y"的形式陈述论据
- 检验论据是否正确、是否适用于当前论证
- 如果论据有争议，准备辩护

大多数论证不需要显式陈述论据，只在读者可能质疑时才加。

## 边界情况

| 情况 | 处理方式 |
|------|----------|
| 用户只有一个理由 | 建议再找1-2个理由，或承认论证较弱 |
| 证据明显不足 | 建议回到 `/source-collection` 补充 |
| 用户忽略反对意见 | 提醒用户主动承认，增强可信度 |
| 论据过于武断 | 建议添加限定条件（"通常"、"在大多数情况下"） |

## Checkpoint

- 确认观点能用一句话说清
- 确认每个理由都能用"因为"连接到观点
- 确认用户考虑过主要反对意见
