# Report Writing

帮用户规划报告结构，撰写初稿，修改组织架构和论证。

## 步骤

### 1. 规划报告结构

帮用户决定两件事：
- **主要论点放在哪**：导言结尾（读者掌控阅读）还是结论开头（读者跟随你）
- **章节顺序**：从旧到新、从简单到复杂、从无争议到有争议

用分镜表或大纲帮用户排列论证要素。

### 2. 撰写暂时性导言

让用户写四句话：
1. **语境**：关于这个题目，读者已经知道什么
2. **问题**：我们不知道或不了解什么
3. **后果**：不知道这些会有什么损失
4. **答案**：我的研究发现了什么（暂时性的）

这个导言会在修改时重写，但能帮用户开始写作。

### 3. 选择写作方式

问用户偏好哪种：
- **快而杂乱**：先写下来，不担心风格，之后再改
- **慢而清晰**：字斟句酌，每个句子确定后再继续

两种都可以，但提醒用户：初稿不是终稿。

### 4. 修改组织架构

从整体到部分检查：
- 导言是否清楚说明了难题和答案？
- 每个章节是否有明确的论点句？
- 章节顺序是否符合逻辑？
- 承认与回应是否放在合适位置？

如果发现问题，先调整架构再改细节。

### 5. 修改论证

检验每个部分：
- 观点是否明确？
- 理由是否支持观点？
- 证据是否可靠充分？
- 是否回应了主要反对意见？

如果论证有缺陷，先修补再继续。

## 边界情况

| 情况 | 处理方式 |
|------|----------|
| 用户不知道如何开始 | 建议先写暂时性导言，从最容易的部分写起 |
| 用户只堆砌资料 | 提醒用户要有自己的观点和论证 |
| 用户纠结风格 | 建议先用"快而杂乱"方式，之后再改 |
| 用户想抄袭 | 警告风险，建议用自己的话重述 |

## Checkpoint

- 确认用户有明确的主要论点
- 确认用户写了暂时性导言
- 确认用户知道初稿不是终稿
