# Wayfinder

一个朦胧的想法到来了——大到一次 agent 会话装不下，又裹在雾里：从这里到**目的地**的路还看不见。寻路（wayfinding）是找出这条路，不是一头冲向目的地。本技能把路画成 issue tracker 上的一张共享**地图**，然后逐个推进它的**决策 ticket**——所谓决策 ticket，是"答案即一个决定"的问题，不是"待执行的构建切片"——直到路径清晰。

目的地因任务而异，命名它是绘图的第一步，也决定后续每张 ticket 的形状。它可能是一份要交接并迭代的规格、一个要在规划开始前锁定的决策、也可能是一处就地完成的改动（比如数据结构迁移）。地图与领域无关——工程、课程内容，凡是符合上述形状的都行。

issue tracker 约定见 `docs/agents/issue-tracker.md`，分诊标签词汇见 `docs/agents/triage-labels.md`。地图与 ticket 用项目的领域术语（见 `CONTEXT.md`），遵守相关区域的 ADR。

## 规划，不是执行

Wayfinder 默认只**规划**：每张 ticket 解决一个决策，地图在路径清晰时完成——别人去把事情做掉之前，已经没有要决定的东西了。"忍不住想直接做"通常是个信号，说明你已经到了地图边缘，该交接了。某项任务可以在地图的 **Notes** 里另行声明，把执行也带进地图；没有这个声明，就只产出决策，不交付物。

## 按名称引用

每张地图、每张 ticket 都是 issue，因此都有一个**名字**——它的标题。在人读到的所有地方（叙述、地图的 Decisions so far），用名字引用，不要用裸的编号、id 或 slug。一墙 `#42, #43, #44` 没法读；名字一眼能懂。id 和 URL 不消失，但它们要*裹在*名字里面，作为链接，不能取代名字。

## 地图

地图是 issue tracker 上的**单个 issue**，打上 `wayfinder:map` 标签——它是规范产物。它的 ticket 是该地图的子 issue。

地图是**索引**，不是仓库。它列出已经做出的决策，指向存放细节的 ticket；一个决策只住在一个地方——它的 ticket——所以地图从不复述决策，只 gist 一行并链接。

**地图本体、子 ticket、阻塞关系、前沿查询在物理上落在哪，取决于具体的 tracker。** 本仓库的 tracker 约定见 `docs/agents/issue-tracker.md`；如未配置，参考那里的本地 markdown 约定。

### 地图正文

整张地图的低分辨率视图，每次会话只加载一次。**未关闭的 ticket 不在此列出**——它们是开放的子 issue，靠查询找到。

```markdown
## Destination

<到达这张地图终点是什么样子——本次寻路要找到的规格、决策或改动。一两行；每次会话在选 ticket 前都先对齐到这里。>

## Notes

<领域；每次会话都该查阅的 skill；本任务长期有效的偏好>

## Decisions so far

<!-- 索引——每张已关闭 ticket 一行：够判断相关性即可，细节点链接到 ticket 看 -->

- [<已关闭 ticket 标题>](link) — <答案的一行 gist>

## Not yet specified

<!-- 见"战争迷雾"：还在范围内、但暂时没法开 ticket 的雾；随前沿推进逐渐"毕业" -->

## Out of scope

<!-- 见"不在范围内"：被判定在目的地之外的工作；关闭后永不毕业 -->
```

### Ticket

每张 ticket 是地图的**子 issue**；tracker 的 issue id 即它的身份。正文就是那个问题，大小控制在一次 100K token 的 agent 会话能装下：

```markdown
## Question

<这张 ticket 要解决的决策或调查>
```

每张 ticket 带 `wayfinder:<type>` 标签——`research`、`prototype`、`grilling`、`task` 之一（见 [Ticket 类型](#ticket-类型)）。

会话**认领**一张 ticket 的方式，是先把它指派给推进该地图的人——在任何工作之前**先做**——这样并发会话就会跳过它。这个指派*就是*认领本身：一张开放且未指派的 ticket，就是未认领。

阻塞关系用 tracker **原生**的依赖关系表达——这一点很关键，因为它会在 tracker 自己的 UI 里把前沿*可视化*，人不打开地图就能看到什么可以拿。只有当 tracker 缺少原生阻塞关系时，才回退到正文约定。一张 ticket 当所有阻塞它的 ticket 都关闭时**解锁**；**前沿**（frontier）是开放、解锁、未认领的子 ticket——已知区域的边缘。

答案不是正文的一部分——它在解决时才记录（见[推进地图](#推进地图)）。解决过程中产出的素材，从 issue 里链接出去，不要贴进正文。

## Ticket 类型

每张 ticket 要么是 **HITL**——人在环里，*和*一个能为自己说话的人一起做——要么是 **AFK**，由 agent 独自推进。HITL ticket 只能通过这种实时交流来解决；agent 永远不替人那一侧发声（一个自己回答自己问题的 grilling agent，已经违反了这条）。

- **Research**（AFK）：读文档、第三方 API、知识库等本地资源，浮出一个决策正等待的事实。由 `/research` **子 agent** 解决。需要当前工作目录之外的知识时用。
- **Prototype**（HITL）：做一个便宜、粗糙、具体的反应物——大纲、草稿、stub，或通过 `/prototype` 写的 UI/逻辑代码——抬高讨论的保真度。把原型作为素材链接进来。核心问题是"它该长什么样"或"它该怎么表现"时用。
- **Grilling**（HITL）：通过 `/grilling` 和 `/domain-modeling` 一次一个问题地交谈。默认情形。
- **Task**（HITL 或 AFK）：在*决策*能做出之前必须发生的体力活——没什么要决策、原型或调研的，但讨论被它卡住。注册一个服务以便评判其 API、开通权限、搬运数据以便看清形状。这是唯一一种*做事*而非*决策*的类型——它之所以配得上地图，是因为它解锁某个决策，而不是因为它交付目的地。agent 能独自做的就独自做（AFK）；不能就给人一份精确的清单（HITL）。活干完即解决；答案记录做了什么以及由此产生、后续 ticket 会依赖的事实（凭据位置、新 URL、行数等）。

## 战争迷雾

地图**有意**保持不完整：不要画你还看不见的部分。已开 ticket 之外是**战争迷雾**——你判断得出"快来了"、但还钉不死的决策和调查，因为它们悬挂在仍未解决的问题上。解决一张 ticket 会拨开它前方的雾，把现在变得能精确陈述的部分"毕业"成新 ticket——一次一张——直到通往目的地的路清晰、没有 ticket 剩余。

地图的 **Not yet specified** 一节就是这片朦胧视图的落脚处：疑似的问题、以后要重访的区域。它是朝向目的地的、尚未发现的前沿——这里的一切都在范围内，只是还没锐利到能开 ticket。视图有多清楚就写多清楚；它同时是路标，给读地图的协作者指明方向。

**雾还是 ticket？** 判据是：你现在能不能精确地陈述这个问题——*不是*你现在能不能回答它。

- **Ticket**：问题已经锐利——即便被阻塞、暂时动不了。
- **Not yet specified**：你还陈述不到那么锐利。不要预先把雾切成 ticket 大小的小块：它比 ticket 粗糙，一片雾在前沿到达后可能毕业成好几张 ticket，也可能一张都不毕业。

**Not yet specified** 排除已经决定的（Decisions so far）、已经是开放 ticket 的、以及不在范围内的（见下一节）。

## 不在范围内

雾只朝目的地聚拢。目的地固定了范围，所以范围之外的工作是**不在范围内**——它不是雾，不属于 **Not yet specified**。它有自己的 **Out of scope** 一节：你刻意判定在*本次*任务之外的工作。是范围、不是锐利度，把它放在这里。

不在范围内的工作永不毕业——前沿停在目的地——所以它只有在目的地被重新画定时才会回来，而且作为一项新任务，不是续接。

把某件事判定为不在范围内，是范围动作，不是路径上的一步。当一张已存在的 ticket 发现自己其实坐在目的地之外——绘图时被错放进来、或被某个解决方案暴露——**关掉它**（关闭的 ticket 毫无歧义地离开了前沿），并在 **Out of scope** 留一行：gist 加上为什么不在范围内，链接到那张关闭的 ticket。它不进 **Decisions so far**——那一节记录实际走过的路，而范围边界不是路上的一步。

## 调用

两种模式。无论哪种，**每次会话最多解决一张 ticket**——research ticket 是例外。

### 绘制地图

用户带一个朦胧想法来调用。

1. **命名目的地。** 跑一轮 `/grilling` 和 `/domain-modeling`，钉死这张地图要寻路到哪里——规格、决策或改动。目的地固定范围，所以最先确定。
2. **勘测前沿。** 再 grill 一轮，这次**广度优先**：在整个空间扇形展开，而非在任何一条线上深挖，浮出开放的决策和当下就能走的第一步。**如果这一步没浮出任何雾**——通往目的地的路已经清晰，整趟旅程小到一次会话装得下——你不需要地图。停下，问用户想怎么继续。
3. **创建地图**（标签 `wayfinder:map`）：Destination 和 Notes 填好，Decisions so far 留空，雾草拟进 **Not yet specified**。
4. **创建现在就能精确陈述的 ticket**，作为地图的子 issue——然后**第二趟**再连阻塞边（issue 要先有 id 才能互相引用）。连线把它们排成前沿和阻塞两组；还没法精确陈述的留在雾里——**Not yet specified** 一节。
5. **启动 research 子 agent。** 对刚创建的每张 `research` ticket，起一个 `/research` 子 agent 并行解决，把发现记在临时的 `research/<name>` 分支上，并在 ticket 里留一个上下文指针。
6. 停下——绘制地图是一次会话的活；它不亲手解决任何 ticket。

### 推进地图

用户带一张地图（URL 或编号）来调用。ticket 是**可选的**——没有就你自己挑下一个决策，不要替用户挑。

1. 加载**地图**——低分辨率视图，不是每张 ticket 的正文。
2. 选 ticket。用户指了哪张就用哪张。否则按顺序取第一张前沿 ticket。**认领它**：在任何工作之前先指派给自己。
3. 解决它——**按需放大**：拉取任何相关或已关闭 ticket 的完整正文；调用 `## Notes` 里点名的 skill。拿不准就用 `/grilling` 和 `/domain-modeling`。
4. 记录解决方案：把答案作为**resolution comment** 发出，**关闭** issue，并在地图的 Decisions so far 里**追加一个上下文指针**。
5. 加入新浮出的 ticket（先创建再连线）；把答案已让其变得可精确陈述的雾毕业出来，把每片毕业的雾从 **Not yet specified** 清掉，让它只作为新 ticket 存在。如果答案暴露某张 ticket——本张或他张——其实坐在目的地之外，**判定为不在范围内**，而非当作路径上的一步解决。如果决策让地图其他部分失效，更新或删除那些 ticket。

用户可能并行跑解锁的 ticket，所以预期会有别的会话同时编辑 tracker。
