# 解决 merge/rebase 冲突

1. **看清当前状态**。检查 merge/rebase 进度、git 历史、冲突文件。

2. **理解冲突双方的意图**。读冲突涉及的 commit message，如果不够清楚，查对应的 PR 描述或 issue。找不到上下文时，从代码改动本身推断意图（变量重命名、逻辑重构、新增功能等）。

3. **逐 hunk 解决**。能同时保留两边意图就保留；不可调和时，选符合本次合并既定目标的那一边，并记下取舍。冲突解决应保留或选择已有改动，不引入冲突双方都没有的新逻辑。**无法判断意图或不确定取舍影响时，停下向用户说明冲突双方的改动和你的顾虑，等确认再继续**。

4. **跑项目的自动化检查**。跑项目的自动化检查（typecheck、测试、lint、格式化等，按项目惯例的顺序）。修掉合并引入的错误。

5. **收尾**。stage 全部改动并提交。rebase 未完成时继续推进，直到所有 commit 都处理完。

## 边界情况

| 情况 | 处理方式 |
|------|---------|
| 不在 merge/rebase 状态（误调用） | 告知用户当前不在冲突解决状态并退出 |
| 冲突文件超过 10 个 | 先列清单让用户确认解决策略 |
| 自动化检查失败且无法修复 | 停下说明情况，列出失败的检查和错误信息 |
| 找不到 commit message/PR 上下文 | 基于代码改动本身判断意图（重命名、重构、新增功能等） |

## 下一步

- 冲突解决完：再跑一次 `/code-review` 确认合并后的改动合理
- 合并后改动通过 review：提交用 `/git-commit`
