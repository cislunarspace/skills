# Implement

按用户在 spec 或 ticket 中描述的内容实现这段工作。

## 核心行为

执行完整的实现循环：

1. **理解需求**：读取 spec 或 ticket，确认要实现的功能、约束条件、验收标准。
2. **设计实现方案**：阅读相关代码，确定实现路径，识别需要修改的文件和模块。
3. **编写代码**：按测试先行的方式推进，先写测试、看测试失败、写实现、看测试通过。每完成一个功能点或模块时，跑相关的类型检查和测试文件。
4. **验证**：全部写完后跑一遍完整测试套件和类型检查，确保所有测试通过、类型检查通过。
5. **提交**：用 `git commit` 将改动提交到本地当前分支。

## Checkpoint

- **开始实现前**：确认已理解 spec/ticket 的需求、约束、验收标准。不清楚时停下问用户。
- **遇到 spec 模糊或矛盾**：停下，向用户说明哪里模糊或矛盾，提出可能的理解，等用户确认后继续。
- **发现需求范围变化**：spec 之外的功能、架构调整、新增依赖，停下说明变化和理由，等用户同意后继续。
- **测试失败或类型检查不通过**：不停下，自己诊断问题、修复代码，直到通过。
- **没有测试框架**：不停下，自己设置标准测试框架（根据语言生态选择），然后继续。
- **提交代码前**：不停下，验证通过后直接提交到当前分支。

## 边界情况

| 情况 | 处理方式 |
|------|---------|
| spec 不清晰或矛盾 | 停下，说明问题，提出可能的理解，等用户确认 |
| spec 要求超出当前技术栈 | 停下，说明技术限制，提出替代方案或需要引入的依赖，等用户同意 |
| 测试失败 | 诊断根因，修复代码或测试，不停下 |
| 类型检查不通过 | 修复类型错误，不停下 |
| 发现既有代码有 bug | 在本次改动范围内就修，超出范围就记录但不修 |
| 没有测试框架 | 设置标准测试框架，不停下 |
| 实现中发现需求范围需要扩大 | 停下，说明为什么需要扩大、扩大多少，等用户同意 |

## 完成条件

- 所有测试通过
- 类型检查通过
- 代码已用 `git commit` 提交到当前分支
- 实现符合 spec/ticket 中的需求和验收标准
