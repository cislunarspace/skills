在当前目录建立教学工作区，根据用户的任务设计和交付自包含的 HTML 课程，跨会话持续推进学习进度。

## 工作流

### 首次调用

1. 检查 `MISSION.md` 是否存在
2. 不存在时，向用户追问学习目标和原因，创建 `MISSION.md`
3. 检查 `RESOURCES.md` 是否有足够的高质量资源，不足时先补充资源
4. 根据学习记录推算用户的最近发展区
5. 设计一节课程
6. 创建共享样式表（`./assets/shared.css`）
7. 交付课程 HTML，可能的话用 CLI 命令打开
8. 写学习记录

### 后续调用

1. 读 `MISSION.md` 和学习记录
2. 检查 `RESOURCES.md` 是否充实，不足时先补充
3. 推算最近发展区（或遵循用户明确指定的内容）
4. 从 `./assets/` 复用现有组件设计课程
5. 交付课程 HTML，可能的话用 CLI 命令打开
6. 写学习记录

## Checkpoint

在以下情况停下来询问用户：

1. `MISSION.md` 不存在或用户学习目标不清晰
2. 需要改动 `MISSION.md` 中的任务
3. 当前目录已有其他项目文件，不确定是否应该建立教学工作区

其他情况自行设计和交付课程。

## 边界情况

| 情况 | 处理方式 |
|------|---------|
| 当前目录不是空的且已有其他项目文件 | 询问用户是否在此建立教学工作区，或切换到其他目录 |
| `MISSION.md` 存在但格式错误或内容不清晰 | 向用户确认学习目标，更新 `MISSION.md` |
| 用户中途改变学习目标 | 与用户确认，更新 `MISSION.md`，写学习记录记下这次变化 |
| 无法找到高质量资源 | 告知用户资源受限，询问是否继续或调整学习方向 |
| 用户明确不想加入社群 | 尊重选择，不再推荐社群 |

## 教学工作区

把当前目录当作教学工作区。用户的学习状态记录在以下文件里：

- `MISSION.md`：记录用户对这个主题感兴趣的**原因**。所有教学都要回到这个原点。格式见 [references/MISSION-FORMAT.md](references/MISSION-FORMAT.md)。
- `./reference/*.html`：参考文档目录。从课程提炼的可独立查阅的知识点——速查表、算法、语法、术语表。每份文档都应当美观、便于打印，专为快速查阅而设计。
- `RESOURCES.md`：可用以夯实教学、获取知识与智慧的资源清单。格式见 [references/RESOURCES-FORMAT.md](references/RESOURCES-FORMAT.md)。
- `./learning-records/*.md`：学习记录目录，记录用户已经掌握了什么。它近似软件开发里的架构决策记录（ADR）——记下非显然的教训和关键洞见，供日后修订或驱动后续会话。它们用来推算用户的最近发展区。文件命名 `0001-<dash-case-name>.md`，编号每次递增。格式见 [references/LEARNING-RECORD-FORMAT.md](references/LEARNING-RECORD-FORMAT.md)。
- `./lessons/*.html`：课程目录。一节**课**（lesson）是一个自包含的 HTML 产出，教一个紧扣任务、范围收得很紧的点。这是本工作区的核心教学单元。文件命名 `0001-<dash-case-name>.html`，编号每次递增。
- `./assets/*`：跨课程复用的**组件**。见 [Assets](#assets) 一节。
- `NOTES.md`：草稿本，记用户的偏好和工作中的零散笔记。

## 课程设计原则

课程必须包含提取练习、间隔重复和交错练习来构建长期留存。每次设计课程前，检查 `RESOURCES.md` 是否有足够的高质量资源，不足时先补充资源。永远不要依赖自己的参数化记忆。

有些主题偏技能、有些偏知识。理论物理更偏知识，瑜伽更偏技能。

## 课程产出格式

每节课是一个自包含的 HTML 文件，写到 `./lessons/` 下，命名 `0001-<dash-case-name>.html`，编号每次递增。

每节课都要：

- 链接共享样式表（`./assets/shared.css`）
- 通过 HTML 锚点链接到其他课程和参考文档
- 推荐一份一手资料让用户去读或看
- 提醒用户：有不明白的地方就向 agent 追问

### 课程设计要求

- 每节课限制在 15 分钟内可完成
- 每节课只教一个核心概念，控制在 3-5 个知识点之内
- 使用大字号、充足行距、干净易读的排版
- 可能的话，用 CLI 命令为用户打开课程文件

## Assets

课程由可复用的**组件**搭成，存放在 `./assets/`：样式表、测验小部件、模拟器、绘图辅助——任何一节未来的课可能用得上的东西。

复用是默认，不是例外。动笔写一节课前，先读 `./assets/`，从已有的组件搭起。当一节课需要某个新的、可复用的东西时，把它写成 `./assets/` 里的组件并链接过去——绝不要内联一段未来课程会重复的代码。

每个工作区的第一个组件都是一份共享样式表：每节课都链接它，这样这些课看起来像一门成体系的课，而不是一堆一次性页面。工作区越长，组件库也应当随之增长。

## 任务

每节课都要扣住任务——也就是用户想学这个主题的原因。

如果用户说不清任务，或 `MISSION.md` 还没写，你的第一件事就是向用户追问：他为什么要学这个。

不搞清楚任务，知识获取就失去了现实目标的锚。课程会变得太抽象，你也没法判断用户下一步该学什么。

任务会随着用户的技能和知识增长而变化，这很正常——与用户确认后更新 `MISSION.md`，并写一条学习记录记下这次变化。

## 最近发展区

每节课，用户都应当觉得难度"恰到好处"——刚好够挑战。

用户可能明确指定想学的东西。如果没有，按下面的方式推算他的最近发展区：

- 读他的 `learning-records`。
- 根据他的任务，判断该教他什么。
- 教落在最近发展区内、最相关的那一样。

## 知识

课程应当围绕一个用户即将掌握的**技能**来设计。课内的知识只放掌握这个技能所必需的部分。先讲知识，再通过交互式的反馈循环让用户练习技能。

知识要先从可信的资源里采集。用 `RESOURCES.md` 追踪它们。课程里要遍布**引用**——把每一个论断都链接到外部资源来支撑。这能提升课程的可信度。

知识讲解使用简单语言，避免同时引入多个新概念。

## 技能

技能通过交互式课程来教。手头有几样工具：

- 用小测验和轻度浏览器内任务的交互式课程。
- 带用户走一组现实世界步骤的课程（比如一组瑜伽体式）。

每一种都要建立在**反馈循环**之上：用户就自己的表现得到反馈。这个循环要尽量紧——立即反馈，最好自动。

测验的每个答案字数应当完全一样（可能的话字符数也一样），避免用答案长度给出提示。

## 社群推荐

智慧来自真正的现实世界互动——在学习环境之外检验自己的技能。当涉及行业经验、个人偏好、实战技巧的问题时，应当推荐用户加入社群（线上或线下的论坛、subreddit、现实中的课程、或本地的兴趣小组）。

你应当去找用户可以加入的高声誉社群。如果用户明确表示不想加入社群，尊重他。

## 参考文档

创建课程的同时，你也应当创建参考文档。课程可以引用这些文档——它们用来沉淀跨课程复用的原始知识单元。

课程很少会被回头翻看；参考文档会。它们应当是课程的压缩精华，专为快速查阅设计。

有些主题天然适合做参考：

- 编程的语法和代码片段
- 流程的算法和流程图
- 瑜伽的体式和序列
- 健身的动作和训练计划
- 任何自有其一套术语的主题的**术语表**

术语表尤其重要。一旦建好，每节课都要遵守它的用语。格式见 [references/GLOSSARY-FORMAT.md](references/GLOSSARY-FORMAT.md)。

## `NOTES.md`

用户有时会表达他对教学方式的偏好，或提醒你某些需要注意的事。这里就是记录这些偏好的地方，便于你在设计课程或与他协作时回头查阅。

## 完成条件

### 单次会话

交付一节课程 + 更新学习记录后，单次会话完成。

### 整个任务

用户明确表示已掌握或不再继续时，整个学习任务完成。
