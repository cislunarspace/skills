# MISSION.md 格式

`MISSION.md` 放在工作区根目录，记录用户学习这个主题的**原因**。每一个教学决策——下一步教什么、推荐哪些资源、设计怎样的练习——都应当能追溯回这份文档。

## 模板

```md
# 任务：{主题}

## 为什么
{1-3 句。用户在追的那个具体的现实目标。掌握这项技能后，他的生活或工作会发生什么变化？避免"为了理解 X"这类抽象表述，要逼出底下的那个结果。}

## 成功长什么样
- {一件用户能做到的、具体可观察的事}
- {另一件具体的事}
- {…}

## 约束
- {时间、预算、已有承诺、学习偏好，任何框定做法边界的因素}

## 不在范围内
- {用户明确表示此刻不想碰的相邻主题——保护最近发展区}
```

## 规矩

- **一个工作区一个任务。** 如果用户想学两件不相关的事，那是两个工作区。
- **具体胜过抽象。** "十月前能跑完半马"胜过"练得更好一点"；"给团队交付一个 Rust CLI"胜过"学 Rust"。
- **遇到模糊要追问。** 如果用户说不清为什么，先访谈他，再动笔。一个糟糕的任务比没有任务更糟。
- **现实变了就改。** 任务会变。用户的目标挪了，就更新这份文件——别让一个过时的任务继续牵引未来的会话。
- **保持简短。** `MISSION.md` 超过一屏，它就不再是罗盘，而是一份计划了。
