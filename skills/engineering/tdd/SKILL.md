# 测试驱动开发

TDD 就是红 → 绿的循环。这个 skill 是让那个循环产出值得保留的测试的参照：什么是好测试、测试该落在哪、有哪些反模式、循环本身的规矩。

按红 → 绿循环实现功能或修复 bug。每个循环：先写一个失败的测试（红），再写刚好让它通过的最小实现（绿），重复直到功能完成。

## 什么是好测试

好测试验证公开接口的行为，不依赖实现细节。判断标准：内部实现完全重写后，测试仍然通过且断言仍然有意义。测试名应该描述能力或规约，如 `valid_cart_can_checkout`，而不是实现步骤。

具体示例见 [references/tests.md](references/tests.md)（好测试与坏测试的对比）和 [references/mocking.md](references/mocking.md)（mock 的使用边界）。遇到测试写法疑问时参考。

## 切面——测试落在哪

**切面**是你进行测试的公开边界：能观察行为、又不必把手伸进内部的接口。测试落在切面上，永远不直接对内部下手。

**只在预先商定的切面上测试。** 写任何测试之前，把要测的切面列出来，跟用户确认。没确认过的切面，不写测试。你没法什么都测——预先约定切面，才能让测试精力落在关键路径和复杂逻辑上，而不是散到每个边界情况。

问：「公开接口是什么？该测哪些切面？」

## Checkpoint

每次循环开始前，必须先与用户确认要测试的切面。列出候选切面（公开 API、CLI 命令、HTTP endpoint 等），等用户确认后再写测试。

## 工作流

每个循环按以下步骤执行：

1. **确认切面**：与用户确认要测试的切面，不确认不继续
2. **写失败测试（红）**：写一个测试，预期行为存在但实现不存在
3. **运行测试确认失败**：测试必须因预期原因失败，意外失败要停下排查
4. **写最小实现（绿）**：写刚好让测试通过的代码，不加额外功能
5. **运行测试确认通过**：测试通过后，小步重构（提取函数、改名）
6. **重复 2-5**：继续下一个测试，直到功能完成

## 反模式

- **耦合实现**——mock 内部协作者、测私有方法，或通过旁路验证（查数据库而不是走接口）。特征：行为没变，一重构测试就挂。
- **同义反复**——断言的期望值用代码自身的方式重新算了一遍（`expect(add(a, b)).toBe(a + b)`、手工算出再贴上的快照、常量跟自己相等），所以它构造上就过，永远没法跟代码不一致。期望值必须来自独立的真值源——已知正确的字面量、手算的例子、规约。
- **水平切片**——先把所有测试写完，再写全部实现。批量测试验证的是**想象出来的**行为：你测的是东西的**形状**，而不是用户面对的行为；测试对真实改动变得不敏感；你在还没理解实现之前就先把测试结构定死了。改成**垂直切片**——一个测试 → 一段实现 → 重复，每个测试是一条 tracer bullet，回应上一个循环教给你的东西。

## 循环的规矩

- **先红再绿。** 先写挂掉的测试，再写刚好够它过的代码。不要预判未来的测试，不要加投机性的功能。
- **一次一片。** 每个循环一个切面、一个测试、一段最小实现。
- **小步重构在绿之后。** 小步重构（提取函数、改名）在绿之后立即进行；大范围重构留到一组循环结束后，配合 `/code-review` 一起做。

## 边界情况

| 情况 | 处理方式 |
|---|---|
| 项目没有测试框架 | 询问用户要使用的测试框架，安装后再继续 |
| 测试运行失败但不是预期的失败 | 停下，报告意外失败，不继续写实现 |
| 找不到要测试的公开接口 | 询问用户期望的切面位置 |
| 测试依赖外部服务（数据库、API） | 询问用户是集成测试还是需要 mock |

## 完成条件

功能需求全部覆盖、所有测试通过、没有待办的测试用例（no pending tests）。
