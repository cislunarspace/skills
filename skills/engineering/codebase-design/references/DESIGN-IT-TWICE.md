# Design It Twice

当用户想为选定的深化候选探索多个候选接口时，用这个并行子 agent 的模式。基于 "Design It Twice"（Ousterhout）——你的第一个想法不太可能是最好的。

用 [SKILL.md](../SKILL.md) 的词汇 —— **模块**、**接口**、**接缝**、**适配器**、**杠杆**。

## 流程

### 1. 把问题空间框出来

在起子 agent 之前，为选中的候选写一段面向用户的问题空间说明：

- 任何新接口都得满足的约束
- 它会依赖什么，各属于哪个类别（见 [DEEPENING.md](DEEPENING.md)）
- 一段粗略的示意代码草图，把约束落到实处 —— 不是提案，只是让约束变具体

把这段给用户看，然后立刻进入第 2 步。用户一边读一边想，子 agent 一边并行干活。

### 2. 起子 agent

用 Agent 工具并行起 3 个以上子 agent。每个都得为深化模块产出一个**截然不同**的接口。

每个子 agent 给一份独立的技术简报（文件路径、耦合细节、来自 [DEEPENING.md](DEEPENING.md) 的依赖类别、接缝背后放什么）。简报与第 1 步那段面向用户的问题空间说明相互独立。给每个 agent 一个不同的设计约束：

- Agent 1："最小化接口 —— 入口点最多 1–3 个。每个入口点的杠杆最大化。"
- Agent 2："最大化灵活性 —— 支持多种用例和扩展。"
- Agent 3："为最常见的调用方优化 —— 让默认场景变得微不足道。"
- Agent 4（如适用）："围绕端口与适配器设计，处理跨接缝依赖。"

简报里同时放进 [SKILL.md](../SKILL.md) 的词汇和 `CONTEXT.md` 的词汇，让每个子 agent 的命名既贴合架构语言，又贴合项目的领域语言。

每个子 agent 产出：

1. 接口（类型、方法、参数 —— 加上不变量、顺序、错误模式）
2. 用法示例，展示调用方怎么用
3. 实现在接缝背后藏了什么
4. 依赖策略与适配器（见 [DEEPENING.md](DEEPENING.md)）
5. 取舍 —— 杠杆高在哪、薄在哪

### 3. 呈现与对比

按顺序把各设计呈现出来，让用户逐个消化，然后用文字把它们对比一遍。从**深度**（接口处的杠杆）、**局部性**（变更集中在哪里）、**接缝位置**三个维度对比。

对比之后，给出你自己的推荐：你认为哪个设计最强、为什么。如果不同设计的元素合在一起会更好，就提出一个混合方案。要表态 —— 用户要的是一个有主张的判断，不是一份菜单。
