# 深化

在给定依赖的前提下，怎样安全地把一簇浅模块深化。假定你已熟悉 [SKILL.md](../SKILL.md) 的词汇 —— **模块**、**接口**、**接缝**、**适配器**。

## 依赖分类

评估一个深化候选时，把它的依赖归一类。类别决定了深化后的模块怎么跨接缝测试。

### 1. 进程内

纯计算、内存状态、没有 I/O。永远可以深化 —— 把模块合并，直接通过新接口测试。不需要适配器。

### 2. 本地可替换

有本地替身的依赖（用 PGLite 替 Postgres、用内存文件系统）。替身存在，就可以深化。深化后的模块让替身在测试套件里跑起来测。接缝是内部的；模块的外部接口上不开端口。

### 3. 远程但自有（端口与适配器）

跨越网络边界、属于你自己的服务（微服务、内部 API）。在接缝处定义一个**端口**（接口）。深模块拥有逻辑；传输作为**适配器**注入。测试用内存适配器。生产用 HTTP/gRPC/队列适配器。

推荐措辞的样子：*"在接缝处定义一个端口，生产实现一个 HTTP 适配器、测试实现一个内存适配器，让逻辑落在一个深模块里——即便它跨网络部署。"*

### 4. 真外部（Mock）

你不掌控的第三方服务（Stripe、Twilio 等）。深化后的模块把外部依赖作为一个注入的端口接收；测试提供一个 mock 适配器。

## 接缝纪律

- **一个适配器意味着一个假想的接缝；两个适配器才意味着真的接缝。** 除非至少有两个适配器站得住（通常是生产 + 测试），否则不要引入端口。只有一个适配器的接缝只是个间接层。
- **内部接缝 vs 外部接缝。** 一个深模块除了接口处的外部接缝，还可以有内部接缝（仅对自身实现可见、供自身测试使用）。不要因为测试用了内部接缝，就把它从接口上暴露出去。

## 测试策略：替换而非叠加

- 浅模块上原有的单元测试，一旦深化模块的接口处有了测试，就成了废物 —— 删掉。
- 在深化模块的接口处写新测试。**接口就是测试面**。
- 测试断言通过接口可观察到的结果，不是内部状态。
- 测试要能熬过内部重构 —— 它们描述的是行为，不是实现。一个测试要是实现一改它就得改，它就是在测接口背后。
