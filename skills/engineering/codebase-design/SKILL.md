# Codebase Design

用这套词汇讨论代码设计。所有设计讨论都使用模块、接口、深度、接缝、适配器这些术语，不换成其他说法。设计目标是给调用方杠杆、给维护者局部性、给所有人可测性。

## 词汇表

使用以下术语，不替换成其他说法。

**模块（module）** —— 任何有接口和实现的东西。刻意不分规模：一个函数、一个类、一个包，或一条横跨多层的东西，都算。_避免_：单元、组件、服务。

**接口（interface）** —— 调用方要正确使用该模块所必须知道的一切：类型签名，也包括不变量、顺序约束、错误模式、必需的配置，以及性能特征（如首次调用慢后续快、O(n) 复杂度）。_避免_：API、签名（太窄——只指类型层面的那一面）。

**实现（implementation）** —— 模块里面的东西，它的代码体。与**适配器**区分开：一个东西可以是小适配器 + 大实现（一个 Postgres 仓库），也可以是大适配器 + 小实现（一个内存 fake）。接缝是话题时用"适配器"，否则用"实现"。

**深度（depth）** —— 接口处的杠杆：调用方（或测试）每学一个单位的接口，能驱动多少行为。一个模块是**深**的，当大量行为藏在一个小接口背后；是**浅**的，当接口几乎和实现一样复杂。

**接缝（seam）** _（Michael Feathers）_ —— 不必改那个地方就能改动行为的地方；模块接口所在的*位置*。接缝放哪本身是一项独立的设计决策，与接缝背后放什么分开。_避免_：边界（与 DDD 的限界上下文混淆）。

**适配器（adapter）** —— 在接缝处满足某个接口的具体物。描述的是*角色*（填哪个坑），不是内容（里面是什么）。

**杠杆（leverage）** —— 调用方从深度里得到的东西：每学一个单位的接口，能拿到更多能力。一份实现，在 N 个调用点和 M 个测试里都还回来了。

**局部性（locality）** —— 维护者从深度里得到的东西：变更、bug、知识与验证集中在一处，而不是散落到各个调用方。修一次，处处都修好。

## 深与浅

**深模块** = 小接口 + 大量实现：少数几个方法、简单参数，背后藏着复杂逻辑。

**浅模块** = 大接口 + 少量实现（要避免）：很多方法、复杂参数，背后只是透传。

设计接口时问自己：

- 能不能减少方法数？
- 能不能简化参数？
- 能不能把更多复杂度藏到里面？

## 原则

- **深度是接口的属性，不是实现的属性。** 一个深模块内部可以由小的、可 mock 的、可替换的部件组成——它们只是不暴露在接口上。
- **删除测试。** 删除这个模块后，如果调用方只需改 import 路径就能直接调用底层，说明是透传层；如果调用方需要各自实现相同的逻辑，说明模块在集中复杂度。
- **接口就是测试面。** 调用方和测试穿过的是同一个接缝。要是你想测到接口*背后*，这个模块大概形状不对。
- **一个适配器意味着一个假想的接缝；两个适配器才意味着真的接缝。** 没有什么真的跨接缝变化，就不要引入接缝。

## 为可测性而设计

好的接口让测试变得自然：

1. **接收依赖，不要自己造依赖。** 让调用方传入 `paymentGateway`，而不是内部 `new StripeGateway()`。
2. **返回结果，不要产生副作用。** 返回 `Discount` 对象，而不是修改传入的 `cart.total`。
3. **表面要小。** 方法少 = 需要的测试少。参数少 = 测试准备简单。

## 不采纳的提法

- **把深度定义为实现行数与接口行数之比**（Ousterhout）：会奖励往实现里灌水。我们用"深度即杠杆"。
- **把"接口"等同于 TypeScript 的 `interface` 关键字或某个类的公开方法**：太窄——这里的接口包含调用方必须知道的每一个事实。
- **"边界"**：和 DDD 的限界上下文撞了。说**接缝**或**接口**。
