# UI 原型

在同一路由下生成**几套截然不同的 UI 变体**，用浮在底部的切换条切换。用户在浏览器里来回翻，挑一个（或从每套里偷一点），其余扔掉。

如果问题关乎逻辑/状态而非东西长什么样——走错分支了，用 [LOGIC.md](LOGIC.md)。

## 什么时候用这条

- "这个页面该长什么样？"
- "定下来之前，我想先看看这个仪表盘的几个方案。"
- "设置页换个布局试试。"
- 任何用户本来要在脑子里花一天在三份模糊 mockup 之间纠结的场景。

## 两种子形状——优先用子形状 A

一个 UI 原型，当它**贴着 App 的其余部分**时（真实的 header、真实的侧边栏、真实的数据、真实的密度），判断起来容易得多。一个孤立的一次性路由是真空：每套变体单独看都不错。只要有合理的现有页面能承载变体，默认走子形状 A。只有当原型确实没有邻近的归属时，才用子形状 B。

### 子形状 A —— 对现有页面的改造（首选）

路由已存在。变体渲染在**同一路由**上，用 `?variant=` URL 查询参数切换。现有的数据获取、参数、鉴权都保留——只换渲染。这是默认，除非有明确理由否则选它。

如果原型针对的是尚不存在的页面、但*天然会住在某个页面里*（仪表盘的新区块、设置页的新卡片、现有流程中的新步骤）——那仍是子形状 A。把变体挂载进宿主页面。

### 子形状 B —— 新页面（最后手段）

仅当被原型的东西确实没有现成页面可住时才用——比如一个全新的顶层界面，或一条没法合理嵌入任何地方的流程。

按项目已有的路由约定创建一条**一次性路由**——别另起一套顶层结构。命名上要让人一眼看出是原型（比如路径或文件名里带 `prototype`）。同样用 `?variant=` 模式。

在定下子形状 B 之前，先自检：真的没有现成页面能嵌入吗？空路由会藏住一个填充过的路由会暴露的设计问题。

两种子形状的浮动切换条都一样。

## 流程

### 1. 说清问题并选 N

默认 **3 套变体**。超过 5 就不再是"截然不同"，而成了噪声——封顶在 5。

把计划一行写下来，放在原型位置或文件顶部注释里：

> "设置页的三套变体，通过 `?variant=` 切换，挂在现有的 `/settings` 路由上。"

这样无论用户在场能否给反馈都管用。

### 2. 生成截然不同的变体

逐套起草。每套要满足：

- 页面的目的和它能拿到的数据。
- 项目的组件库 / 样式系统（TailwindCSS、shadcn、MUI、原生 CSS，都行）。
- 清晰的导出组件名，如 `VariantA`、`VariantB`、`VariantC`。

变体必须**结构上不同**——不同的布局、不同的信息层级、不同的主要操作点，不只是颜色不同。三套微调过的卡片网格不是 UI 原型，是壁纸。如果两套草稿出来太像，就明确"不要用卡片网格"重做一套。

### 3. 把它们接到一起

在路由上创建一个单一的切换器组件：

```tsx
// 伪代码 —— 适配项目的框架
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

子形状 A（现有页面）：把所有现有数据获取放在切换器之上；只有渲染的子树随变体变化。

子形状 B（新页面）：`/prototype/<name>` 下的一次性路由挂载同一个切换器。

### 4. 搭浮动切换器

屏幕底部居中一条小固定栏，三部分：

- **左箭头** —— 切到上一套变体（循环）。
- **变体标签** —— 显示当前变体的 key，如果变体导出了名字，也显示名字。如 `B — 侧边栏布局`。
- **右箭头** —— 切到下一套（循环）。

行为：

- 点箭头更新 URL 查询参数（用框架的 router——Next 上 `router.replace`、React Router 上 `navigate` 等），让变体可分享、刷新稳定。
- 键盘：`←` 和 `→` 键也能切换。当 `<input>`、`<textarea>` 或 `[contenteditable]` 获得焦点时不要拦截方向键。
- 视觉上与页面区分（如高对比药丸形、轻微阴影），让人一眼看出它不是被评估设计的一部分。
- 生产构建中隐藏——用 `process.env.NODE_ENV !== 'production'` 或等价检查把关，防止一次 stray 的原型合并把切换条送到用户面前。

把切换器放进一个共享的单一组件，让两种子形状都能复用。放在项目里共享 UI 所在的位置。

### 5. 交接

给出 URL（和 `?variant=` 的 key）。用户想看就看。有意思的反馈通常是**"我想要 B 的 header 配 C 的侧边栏"**——那才是他们真正要的设计。

### 6. 记录答案并清理

一旦某套变体胜出，记下答案——哪套、为什么——然后按 [SKILL](../SKILL.md) 描述的方式留存原型。把胜出者折进真实代码，其余移到 throwaway 分支，不进 main：

- **子形状 A** —— 把胜出者折进现有页面；从 main 移除落败变体和切换器。
- **子形状 B** —— 把胜出变体提升为真实路由；从 main 移除一次性路由和切换器。

全套变体是一手资料，所以落在 throwaway 分支，不是垃圾桶——留在 main 分支里的变体组件和切换器很快会腐烂、迷惑下一个读者。

## 反模式

- **只差颜色或文案的变体。** 那是微调，不是原型。真正的变体在结构上意见相左。
- **变体之间共享太多代码。** 共享一个 `<Header>` 可以；共享 `<Layout>` 就违背初衷了。每套变体应能自由扔掉布局。
- **把变体接到真实 mutation 上。** 只读原型没问题。如果某套变体需要 mutate，指向一个 stub——问题是"这东西该长什么样"，不是"后端能不能用"。
- **直接把原型提升到生产。** 变体代码是在原型约束下写的（无测试、最小错误处理）。折进真实代码时要好好重写。
