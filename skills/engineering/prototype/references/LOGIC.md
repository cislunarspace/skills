# 逻辑原型

一个极小的交互式终端程序，让用户手动驱动一个状态模型。当问题关乎**业务逻辑、状态转换或数据形状**时用它——这类东西在纸上看着合理，只有推过真实用例才发现不对劲。

## 什么时候用这条

- "我不确定这个状态机能不能处理 X 之后 Y 的边界情况。"
- "这个数据模型真能表达……那种情况吗？"
- "写之前我想先摸一下 API 该长什么样。"
- 任何用户想**按按钮、看状态变**的场景。

如果问题是"这东西该长什么样"——走错分支了，用 [UI.md](UI.md)。

## 流程

### 1. 把问题说清楚

写代码之前，先写下你要原型的是什么状态模型、要回答什么问题。一段话，放在原型的 README 或文件顶部注释里。一个回答错了问题的逻辑原型是纯浪费——把问题写明，方便事后核对，无论用户现在在场还是 AFK 回来看。

### 2. 选语言

用宿主项目用的语言。如果项目没有明显的运行时（比如文档仓库），就问。

遵循项目已有的工具约定——别为原型引入新的包管理器或运行时。

### 3. 把逻辑隔离进一个可移植模块

把真正回答问题的那段逻辑，藏在一个小而纯的接口后面，将来能整个拎出来、丢进真实代码库。外围的 TUI 是一次性的；逻辑模块不该是。

正确的形状取决于问题：

- **纯 reducer** —— `(state, action) => state`。当动作是离散事件、状态是单值时合适。
- **状态机** —— 显式的状态和转换。当"现在哪些动作合法"本身就是问题的一部分时合适。
- **一组纯函数** 作用于一个朴素数据类型。当没有隐式的当前状态、只有变换时合适。
- **一个类或模块，有清晰的方法面**，当逻辑确实持有持续的内部状态时。

挑最贴合问题的形状，*不是*最好接到 TUI 上的那个。保持纯净：无 I/O、无终端代码、不用 `console.log` 控制流程。TUI 导入它、调它；没有反向的数据流。

这正是让原型超越自身寿命的关键：问题答完后，验证过的 reducer / 状态机 / 函数集可以单独拎进真实模块。

### 4. 搭最小的、能暴露状态的 TUI

搭一个**轻量 TUI**——每个 tick 清屏（`console.clear()` / `print("\033[2J\033[H")` / 等价写法）并重新渲染整帧。用户看到的应始终是一个稳定视图，而非不断增长的滚动历史。

每帧两部分，按这个顺序：

1. **当前状态**，pretty-print 且对 diff 友好（每行一个字段，或格式化的 JSON）。字段名或小节标题用**粗体**，次要的上下文（时间戳、ID、派生值）用**暗色**。原生 ANSI 转义码就行——`\x1b[1m` 粗、`\x1b[2m` 暗、`\x1b[0m` 重置。除非项目已经用了样式库，不必专门引入一个。
2. **键盘快捷键**，列在底部：`[a] 添加用户  [d] 删除用户  [t] 推进时钟  [q] 退出`。键粗体、描述暗色，或反过来——怎么清楚怎么来。

行为：

1. **初始化状态** —— 一个内存中的对象/结构体。启动时渲染第一帧。
2. **一次读一个按键（或一行）**，派发给一个改动状态的 handler。
3. **每次动作后重新渲染整帧** —— 不是追加，是替换。
4. **循环到退出。**

整帧应在一屏内显示完。

### 5. 一条命令能跑

往项目已有的 task runner（`package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）里加一条脚本。用户应能跑 `pnpm run <原型名>` 或等价命令——永远不必记路径。

如果宿主项目没有 task runner，就把命令写在原型 README 顶部。

### 6. 交接

把运行命令交给用户。他们自己驱动；有意思的瞬间是他们说"等等，这不该可能"或"咦，我以为 X 会不一样"——那些是_想法_里的 bug，正是整个原型的目的。他们要加新动作，就加。原型会演化。

### 7. 记录答案和原型

一旦原型答完了它的问题，记下答案，然后按 [SKILL](../SKILL.md) 描述的方式留存原型。逻辑原型特有的映射：验证过的 reducer / 状态机 / 函数集拎进真实模块（决策被吸收）；TUI 外壳跟着进入那个把原型作为一手资料的 throwaway 分支。

## 反模式

- **别加测试。** 一个需要测试的原型就不再是原型了。
- **别接到真实数据库。** 用内存存储，除非问题本身就是关于持久化的。
- **别泛化。** 不要"万一以后要支持 X"。原型回答一个问题。
- **别把逻辑和 TUI 搅在一起。** 如果 reducer / 状态机引用了 `console.log`、prompt 或终端转义码，它就不再可移植了。把 TUI 当作纯模块外面的一层薄壳。
- **别把 TUI 外壳送进生产。** 外壳是为"从终端手动驱动"优化的。它后面的逻辑模块才值得留下。
