# Prototype

原型是**用来回答某个问题的一次性代码**。问题决定了原型的形状。

## 先分清是哪条路

确定要回答的是哪个问题——从用户的话、周围代码推断，或用户在场就直接问：

- **"这段逻辑 / 状态模型成立吗？"** → 走 [LOGIC.md](references/LOGIC.md)。搭一个极小的交互式终端程序，把状态机推到那些纸上难以推演的边界情况。
- **"这东西应该长什么样？"** → 走 [UI.md](references/UI.md)。在同一路由下生成几套截然不同的 UI 变体，用 URL 查询参数加一个浮在底部的切换条来回切。

两条路产出的东西差别极大，选错整个原型就白费。如果问题确实模糊、又联系不上用户，就按周围代码判断（后端模块 → 逻辑；页面或组件 → UI），并在原型开头注明这个假设。

## 两条路共通的规矩

1. **从第一天起就是一次性的，并明确标注。** 把原型代码放在它将来真正会被使用的位置旁边（紧挨它为之做原型的模块或页面），让背景一目了然——但命名上要让随便翻到的人一眼看出这是原型，不是生产代码。一次性的 UI 路由要遵守项目已有的路由约定，别另起一套顶层结构。

2. **一条命令能跑。** 用项目现有 task runner 支持的方式——`pnpm <name>`、`python <path>`、`bun <path>` 等。用户必须能不加思索地启动它。

3. **默认不落盘。** 状态放在内存里。持久化是原型要_验证_的东西，不是它能依赖的东西。如果问题明确涉及数据库，就打一个临时库或本地文件，取个明显的名字如 `PROTOTYPE — wipe me`。

4. **别打磨。** 不写测试、不做错误处理（除非不处理就跑不起来）、不做抽象。目的是快速学到一件事。

5. **让状态可见。** 每次动作后（逻辑）或每次切换变体后（UI），打印或渲染出完整的相关状态，让用户看清变化。

6. **做完要归档。** 把任何经过验证的决策折回真实代码，然后把原型本身作为**一手资料**留存：提交到一个一次性分支（不进 main），并在实现 issue 上留一个指向该分支的上下文指针。也要记下答案——结论和它解决的问题——写进 issue 或 commit。main 分支只保留验证过的决策。

## 下一步

- 原型验证想法成立：把通过的决策折回真实代码用 `/implement`
- 原型验证想法不成立：丢掉原型，不需要下一步
