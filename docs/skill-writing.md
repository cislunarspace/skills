# Skill 写作规范

SKILL.md 是 agent 在编码过程中**被触发的行为指令**，不是一次性任务 prompt。写好 skill 的核心是讲清三件事：**做什么、边界在哪、什么时候停下来问**。教模型怎么思考、堆角色设定，没用。

## 和任务 prompt 的区别

常说的“好 prompt”五要素（Context / Request / Output Format / Constraints / Checkpoint）针对的是**给模型派一次活**。Skill 是长期挂载、等调用的规范，Context 和 Output Format 每次调用都不同，写死进 SKILL.md 是错的。映射如下：

| 任务 prompt 要素 | 在 skill 里怎么处理 |
|---|---|
| Context | 不写。调用时的上下文是动态的。触发信息放进 frontmatter 的 `description` |
| Request | 正文开头写**核心行为** |
| Output Format | 只在 skill 有固定产出时写（如 commit message 格式、交接文档结构） |
| Constraints | **边界情况表** + 明确的“不要做什么” |
| Checkpoint | 写明**什么时候停下来问用户、什么时候自己继续** |

## frontmatter：调度的依据

模型靠 frontmatter 决定要不要调用这个 skill，所以这里只服务于一件事——**让调度判断准确**。

```yaml
---
name: <skill-name>                   # 目录名，也是命令名（/<skill-name>）
description: <一句话做什么>。当用户<触发场景/触发词>时使用。
argument-hint: "可选的参数提示"
disable-model-invocation: true      # 可选：只能手动触发，模型不会自动调用
---
```

- **`name`** = 目录名。命令名直接由它来（`<name>` → `/<name>`）。
- **`description`** 写两件事：**做什么**（一句话）+ **什么时候用**（触发场景、触发词）。模型靠后半句决定是否调用。只写“是什么”不够，要写“什么情况下该轮到我”。
- **`argument-hint`**（可选）：参数提示，告诉用户调用时可以传什么。
- **`disable-model-invocation`**（可选）：设 `true` 的 skill 只能由用户显式发起，模型不会自动触发。需要人主动发起的 skill 用它。

## 正文：按需写这几节

不是每节都要。按 skill 的复杂度挑着写。

### 核心行为（必须有）

开头一段话或几行祈使句，说清这个 skill 执行什么动作。极简的 skill 整个正文就是核心行为，十几行讲清楚就够了：

> 就这个计划的每个方面对我进行不间断的追问，直到达成共识。……一次只问一个问题，等拿到反馈再继续。如果一个问题可以通过探索代码库来回答，就去探索代码库，不要问用户。

### 步骤 / 工作流（多步时）

有明确流程的 skill 分步骤写，每步是**可检查的**动作。典型如分四步走（分析起草 → 确认执行 → 验证 → 收尾），关键步骤结尾标“完成条件”。

### 边界情况（和外部世界打交道时）

skill 涉及文件、git、网络、用户输入时，用表格列**情况 → 处理方式**。这是 skill 里最实用的一节——模型遇到意外时照表办事，而不是自己猜。范例：

| 情况 | 处理方式 |
|---|---|
| 没有会话文件被改动 | 不提交，告知用户 |
| 会话文件存在合并冲突 | 不提交，先解决冲突 |

### Checkpoint（涉及确认时）

写明哪些地方**必须停下来等用户**，哪些**自己做完整再说**。涉及不可逆操作的 skill 要写死“绝不自行执行”；纯确定性、无副作用的脚本型 skill 不用停。

只有三种情况该暂停：

1. 涉及不可逆操作（提交、推送、删除、关闭 issue）
2. 任务范围发生变化（用户要的东西和最初不一样了）
3. 需要用户提供信息才能继续

其他情况自己做完再汇报。

### 完成条件（有明确终点时）

做成什么样算完成，逐条列出（如退出码 0、内容逐字一致、LF 行尾）。没有明确终点的 skill（交互式、纪律型）不要硬加——它们本来就没有终点。

## 不该写的

- **角色设定**（“你是一个资深工程师，擅长……”）。模型不需要被赋予身份。
- **教模型怎么思考**（“以批判性思维审视”“仔细想想”）。写“做什么”和“边界在哪”，不写“用什么心态”。
- **死的 Context / Output**。调用时才有的背景，别写进 SKILL.md。
- **为完整而注水**。没有边界情况就别编边界情况表，没有参数就别写 `argument-hint`。

## 长度

短的 skill 十行以内，长的近百行，都合理。判断标准不是行数，是**每一段是否改变了模型的行为**：

> 删掉这一段后，agent 会做出不同的事吗？不会，就删掉。

## 起步

新 skill 从 [`templates/SKILL.md`](./templates/SKILL.md) 复制骨架，按需删节，参照本仓库现有 skill 的风格。
